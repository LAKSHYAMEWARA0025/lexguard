# LEXGUARD Context

## Overall Goal
Building LEXGUARD: an adversarial, multi-agent AI contract intelligence platform.

## Tech Stack
- **Frontend/Backend:** Next.js (App Router, TypeScript, Tailwind CSS)
- **Database/Auth:** Supabase
- **Orchestration:** LangGraph (Agents)
- **Caching/State:** Upstash (Redis)
- **Media/Assets:** Cloudinary

## Progress So Far
- **Initialization:** Core Next.js project initialized with required dependencies.
- **Base Layout & UI:** Created the foundational folder structure (`agents/`, `lib/`, `components/`) and implemented a responsive, professional Hero component for the landing page with a static legal tech background image.
- **Hydration Fix:** Added `suppressHydrationWarning` to the root HTML layout to prevent browser extension mismatch errors.
- **Database & Storage Configuration:** Scaffolded `.env.local` for credentials, initialized `lib/supabase.ts` and `lib/cloudinary.ts` storage clients, and created the initial Supabase pgvector schema migration. The `document_chunks` table and `match_document_chunks` RPC function are now configured for `vector(3072)` to accommodate `models/gemini-embedding-001`. We are intentionally *not* using an HNSW index because pgvector restricts it to 2000 dimensions, and exact nearest-neighbor search is perfectly performant for document-scale RAG.
- **Phase 2: LangGraph Orchestration Begun:** Defined our `GraphState` schema using the LangGraph Annotation pattern (`src/agents/state.ts`), managing state for documentId, queries, retrievedChunks, risks, and finalReport. Implemented the first agent node, `queryExpander` (`src/agents/nodes/queryExpander.ts`), leveraging `gemini-2.5-flash` (updated from `gemini-1.5-pro` to fix a 404 API error) and `zod` to force structured JSON outputs containing strategic legal search queries. Created the LangGraph compilation engine (`src/agents/graph.ts`) mapping the nodes and edges, and built a test API route (`/api/analyze`) to execute the graph pipeline.
- **Phase 2: Retriever Node:** Completed the `retrieverNode` (`src/agents/nodes/retriever.ts`), which validates state and concurrently maps generated queries to 3072-dimension vectors (`models/gemini-embedding-001`). It executes concurrent Supabase RPC searches (`match_document_chunks`) and implements Map-based deduplication to ensure unique context chunks are passed downstream. The `match_document_chunks` RPC function was explicitly updated to accept a `filter_document_id` UUID and enforce it in the SQL `WHERE` clause to guarantee multi-tenant document isolation during RAG retrieval.
- **Phase 2: Adversarial Red Team Node:** Implemented `redTeam` (`src/agents/nodes/redTeam.ts`) leveraging `gemini-2.5-flash` with a highly aggressive "vulture lawyer" persona. It analyzes retrieved chunks for hidden fees, one-sided clauses, and missing protections, outputting structured JSON via a rigid Zod schema (capturing severity, clause snippet, plain-English issue, and recommendation). Wired the graph to flow: `queryExpander` -> `retrieverNode` -> `redTeam` -> `__end__`.
- **Phase 2: Verifier Node:** Implemented `verifierNode` (`src/agents/nodes/verifier.ts`) to serve as a strict Legal Auditor. It leverages `gemini-2.5-flash` to strictly cross-reference the Red Team outputs against the retrieved text chunks, deleting any AI hallucinations. Corrected the schema to exactly mirror the state shape and rewired the LangGraph engine in `src/agents/graph.ts` to securely place the Verifier between the Red Team and Advisor.
- **Phase 2: Advisor Node:** Implemented the terminal `advisorNode` (`src/agents/nodes/advisor.ts`) functioning as brutally honest senior counsel. It maps raw risks into a strict Zod schema (`advisorReport` and `overallVerdict`) that highlights the "trap", the "harshReality", and provides "advice". Updated `src/agents/state.ts` to strictly type the `finalReport`. The LangGraph workflow is officially wired as: `queryExpander` ➔ `retrieverNode` ➔ `redTeam` ➔ `verifierNode` ➔ `advisorNode` ➔ `__end__`. Standardized, extensive prefix logging (e.g., `[AdvisorNode]`) was implemented across all nodes to track raw execution, prompt inputs, and Zod parsing success/failures.
- **Phase 3: Blue Adversarial HUD (UI Overhaul):** Redesigned the frontend (`src/components/Hero.tsx`) into a futuristic, high-stakes cybersecurity HUD utilizing a "Blue Cyberpunk" aesthetic. Replaced standard fonts with `Space Grotesk` and `JetBrains Mono` in `layout.tsx`. Added a high-resolution, static legal-tech hero image (`public/lexguard_static_hero.png`) placed securely within a strictly non-overflowing main wrapper (`relative min-h-screen w-full overflow-hidden`). Injected `.bg-grid-pattern`, `.glass-panel`, `.neon-border-blue`, and an `.animate-scan` keyframe into `globals.css`. The UI seamlessly transitions from a glowing blue dropzone into a split-screen layout during analysis, simulating rapid terminal logs, before rendering the internally-scrolling "Threat Matrix". A definitive pre-deployment UI fix was implemented to patch a layout breakage preventing scrolling by strictly constraining the parent results container (`h-[85vh] overflow-hidden`) and applying `overflow-y-auto` to the right-side Risk Matrix container. Functional color-coding for risk cards was restored via `getSeverityConfig`: HIGH=Red, MEDIUM=Yellow, LOW=Green borders and accents, restoring immediate visual risk prioritization while maintaining the overall Blue HUD aesthetic.
- **Ingestion Pipeline (Multi-Format):** Built a full ingest flow (`/api/ingest`) that uploads to Cloudinary, extracts text via a multi-format MIME-type router supporting `.pdf` (via `unpdf` with `mergePages: true`), `.docx` (via `mammoth`), and `.txt` files. The frontend UI (`Hero.tsx`) was updated to accept these new formats. The text is chunked via LangChain, vectorized using `models/gemini-embedding-001`, and bulk-inserted into Supabase. Corrected the asynchronous embedding generation and mapping logic with explicit dimension validations to fix a Supabase pgvector dimension error, and added comprehensive debug logging to the `/api/ingest` route.
- **RAG Optimization (Hard/Buried Clauses):** Optimized the ingestion pipeline and strategist agent to catch complex, cross-referenced traps. Increased the `RecursiveCharacterTextSplitter` chunk size to 2000 with a 500-character overlap in `/api/ingest/route.ts` to preserve context across pages. Expanded the `queryExpander` prompt to explicitly generate 5-7 queries targeting advanced liabilities, including AI training data rights, IP ownership, and unilateral modification clauses. To prevent context starvation from the larger chunks, adjusted `src/agents/nodes/retriever.ts` by increasing the `match_count` to 8 and lowering the `match_threshold` to 0.5.
- **API Rate Limit Resilience:** Implemented an exponential backoff utility (`src/lib/withRetry.ts`) to wrap all LLM invocations across the LangGraph nodes (`queryExpander`, `redTeam`, `verifier`, `advisor`). This ensures the system automatically intercepts and recovers from Gemini API 429 Rate Limit errors without crashing. Updated the `/api/analyze/route.ts` endpoint to intercept internally failed graph runs and gracefully propagate a 429 status code back to the frontend to inform the user of system overload.
- **Pre-Deployment Audit:** Completed a stability and security audit prior to launch. Implemented a strict 4.5MB file size limit in the ingestion route (`/api/ingest/route.ts`) to comply with Vercel's serverless function memory constraints and prevent 500 crashes. Swept all API routes to ensure no environment variables or credentials were inadvertently logged to the terminal.
- **Autonomous Classifier Node:** Implemented `classifierNode` (`src/agents/nodes/classifier.ts`) as the new entry point for the LangGraph pipeline. It fetches the very first document chunk from Supabase to infer the `documentContext` (e.g., "SaaS Subscription Agreement between Vendor and Client") via `gemini-2.5-flash`. This context is dynamically injected into the `queryExpander` node, ensuring the adversarial RAG queries are hyper-targeted to the specific contract type without requiring any manual user input.
- **Cloudinary DNS/EAI_AGAIN Fix:** Resolved a regression causing `EAI_AGAIN` on `api.cloudinary.com` during ingestion by ensuring `cloudinary.config` uses `process.env.CLOUDINARY_CLOUD_NAME` as defined in `.env.local`. Enforced `export const dynamic = 'force-dynamic';` in `/api/ingest/route.ts`, added a "quiet" server-side check to proactively catch missing environment variables before stream upload, and verified the frontend `fetch` omits the `Content-Type` header when passing `FormData`.
